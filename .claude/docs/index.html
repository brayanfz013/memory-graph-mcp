Overview

memory-graph-mcp is a unified semantic memory system for AI agents. It provides one MCP (Model Context Protocol) server that gives Claude Code, Codex, Cursor, or any MCP client a per-repo knowledge store with four reinforcing layers:

📊
Vector Memories
Findings with embeddings for semantic recall of past solutions and decisions.
🕸
Knowledge Graph
Typed nodes + edges (SOLVES, SUPERSEDES...) + PageRank to surface what matters most.
📖
Wiki
Long-form curated docs auto-generated from canonical KG nodes.
🤝
Collective + Cache
Cross-agent state with TTL to coordinate multi-agent work without re-running expensive tools.
Key design principle: Storage is workspace-scoped DuckDB at <repo>/.memory-graph/memory.duckdb — no cloud, no API keys for local embeddings (uses fastembed with BGE models).

Storage Layer

Everything lives in a single DuckDB file with the VSS (Vector Similarity Search) extension for HNSW-based vector search using cosine similarity.

DuckDB advantages: Single-file database, WAL-based concurrency (no lock conflicts under normal use), built-in HNSW index, no external dependencies. The VSS extension provides native vector operations like array_cosine_similarity().

PageRank

PageRank is computed iteratively over the KG edges. It scores nodes by importance based on the graph structure:

For each iteration:
  new_score[target] = (1-damping)/N + damping * sum(
    source_score[source] * weight(edge) / out_weight[source]
    for all sources pointing to target
  )

Convergence: stop when max diff < 1e-6 (typically 10-30 iterations)

The damping factor defaults to 0.85 (standard value). PageRank is re-computed after memory_consolidate(dry_run=False).

BFS Expansion

When recall(hops=N) is called, seed nodes found by semantic search are expanded N hops via breadth-first search to surface related context:

hops=0: Only semantic seed nodes
hops=1: Seeds + their immediate neighbors (edges)
hops=2: Seeds + 1-hop + 2-hop neighbors
...

Tool Surface

Primary Tools (use these by default)

recall(query, scope, top_k, min_score, node_type, hops) ▶

Unified semantic search across all three backends. This replaces the old trio of memory_recall / kg_query / wiki_query.

Parameters

  • query: Natural language search string
  • scope: memories | kg | wiki | all (default)
  • top_k: Max results per scope
  • min_score: Cosine similarity floor (0-1), default 0.45
  • node_type: Optional KG filter (Decision, Solution, etc.)
  • hops: BFS expansion from KG seed nodes (0 = disabled)

Return shape

{
  "query": "...",
  "scope": "all",
  "memories": [{id, type, content, score, metadata, ...}],
  "kg": {
    "nodes": [{node_id, node_type, label, semantic_score, pagerank_score, ...}],
    "edges": [{from_id, to_id, rel_type, weight}],
    "seed_count": 5,
    "expanded_count": 3
  },
  "wiki": [{page_id, canonical_id, title, snippet, score, tags}],
  "top_canonicals": ["solution.duckdb-lock-retry", ...],
  "search_mode": "semantic"
}

How it works internally

  1. Embeds the query vector via the configured embedding provider
  2. For memories: HNSW nearest neighbor search over memory_embeddings with cosine similarity
  3. For kg: Joins memory embeddings → memories → kg_nodes (via properties_json memory_id), then falls back to ILIKE on label/properties if embeddings yield insufficient results. Increments reuse_count on hits. Expands via BFS.
  4. For wiki: HNSW search over wiki_embeddings with ILIKE fallback
  5. Fuses results into top_canonicals — the most relevant canonical_ids across all scopes (feed to wiki_get)

Auto-edge inference

After creating the new node, it embeds {title}\n{content} and searches for the top-3 semantically similar existing KG nodes (via memory embedding similarity). Creates RELATED_TO edges only if score ≥ 0.62 and no duplicate exists.

Auto-promotion

If the node's reuse_count reaches 3, it auto-promotes from draft to canonical. The reuse_count is incremented every time the node appears in a recall result.

wiki_get(canonical_id_or_title) ▶

Fetch a full wiki page in its entirety. Lookup priority: (1) by stable canonical_id slug, (2) by exact title match (case-insensitive). Returns the complete body plus tags, author, timestamps, and linked KG node.

Graph Navigation Tools

kg_neighbors(node_id, direction, rel_type)▶

Get immediate neighbors of a KG node. Filter by direction (outgoing/incoming/both) and relation type. Returns sorted by PageRank.

kg_path(from_id, to_id, max_depth)▶

Find shortest path between two KG nodes via BFS. Returns the node path, edge list, and path length. Undirected traversal.

kg_resolve(canonical_id)▶

Look up a node by its stable canonical_id slug (e.g., solution.duckdb-lock-retry). Returns full node details.

Lifecycle / Advanced Write Tools

kg_promote(node_id, status)▶

Manually transition a node: draft → canonical → superseded. Canonical promotion triggers wiki crystallization on the next record_finding call (or run wiki_bootstrap).

kg_add_edge(from_id, to_id, rel_type, weight)▶

Manual typed edge. Valid relations: SOLVES, CAUSED_BY, DEPENDS_ON, RELATED_TO, USES_TOOL, SUPERSEDES. record_finding auto-infers RELATED_TO edges; use this for typed edges the system can't infer.

wiki_ingest(title, body, canonical_id, tags, author)▶

Manually author a wiki page. Auto-embedded for semantic recall. Most cases should let record_finding crystallize wikis automatically; use this for long-form, hand-written documentation.

Swapping Providers

1. embedding_status()
→
2. embedding_benchmark()
→
3. embedding_migrate(dry_run=True)
→
4. embedding_migrate(dry_run=False)

The key insight: source content is never touched during migration. Only the vectors are recreated with the new model's dimensions. The embedding_meta table tracks which provider/model/dimensions were used for each "generation", so mismatches on startup are detectable and auditable.

Migration Details

embedding_migrate is batched (default 64 texts per batch) and runs as a plan-then-apply workflow:

# Step 1: Plan (safe, no rewrite)
embedding_migrate(target_provider="fastembed", target_model="BAAI/bge-base-en-v1.5", dry_run=True)
# Returns: plan, estimated work, source counts

# Step 2: Apply
embedding_migrate(target_provider="fastembed", target_model="BAAI/bge-base-en-v1.5", dry_run=False)
# Re-embeds all memory + wiki page vectors under the new model

Wiki Layer

The wiki stores human-refined knowledge that outlives ephemeral memory. Each wiki page is linked to a canonical KG node and provides long-form, authoritative documentation.

Crystallization

When a KG node becomes canonical, it auto-crystallizes into a wiki page. The crystallization extracts:

  • Title from node label
  • TL;DR from tldr_32
  • Summary from brief_96 / summary_256
  • Full content from properties_json.content
  • Related files from properties_json.files
  • Tags from properties_json.tags

Wiki Bootstrap

wiki_bootstrap() auto-populates the wiki by scanning the workspace structure in 4 phases:

1. Workspace overviewDir tree + READMEs
→
2. Per-directory pagesREADME + code summaries
→
3. Config files pagepyproject.toml, etc.
→
4. Crystallize canonicalsKG nodes → wiki pages

For Python files, it extracts docstrings, public classes, and public functions. For JS/TS files, it extracts JSDoc comments and exports. Directory pages include the directory tree structure, README content, and code summaries.

Auto-Edges Deep Dive

When record_finding is called, it automatically infers up to 3 RELATED_TO edges by:

  1. Embedding {title}\n{content} of the new finding
  2. Searching for top-3k semantically similar existing KG nodes (via their linked memory embeddings)
  3. Filtering by score threshold (≥ 0.62)
  4. Deduplicating (skips if RELATED_TO edge already exists)
  5. Limiting to 3 edges total

Auto-Promotion Deep Dive

Each time a KG node appears in a recall result, its reuse_count is incremented. When reuse_count reaches 3, the node auto-promotes from draft to canonical. This means a piece of knowledge is considered "canonical" once it has been recalled 3 times — indicating it's frequently relevant and valuable.

Database Schema

The DuckDB schema (v5) with all tables:

memories          — Text memories with type, metadata, TTL
memory_embeddings — Vector embeddings (HNSW index, cosine)
kg_nodes          — Knowledge graph nodes (lifecycle, summaries, PR scores)
kg_edges          — Directed, typed edges (from_id, to_id, rel_type, weight)
collective_memory — Cross-agent shared state (key, scope, type, value, TTL)
tool_cache        — Cached tool call results (tool_name, args_hash, result, TTL)
wiki_pages        — Crystallized wiki pages (body, tags, author, status)
wiki_embeddings   — Wiki page vector embeddings (HNSW index, cosine)
embedding_meta    — Provider identity tracking (generation, provider, model, dim, active)

Concurrency

DuckDB WAL concurrency: DuckDB uses Write-Ahead Logging for concurrent access. Multiple processes can read/write simultaneously without conflicts under normal conditions.

Additional protections:

  • @with_retry decorator: All database operations are decorated with exponential backoff retry (3 attempts, 0.2s base delay, 2s max) for transient lock errors
  • Single connection per operation: get_connection() opens and closes a connection per call
  • HNSW index rebuild: Every 50 inserts, the HNSW index is rebuilt (not per-insert, to avoid overhead)
  • Singleton VSS extension: Installed once per process (_vss_installed flag)
  • Singleton schema init: Run once per process (_schema_ready flag)

Embedding Provider Interface

All providers implement a uniform contract:

class Provider:
    provider: str           # "fastembed" | "ollama" | "vertex"
    model_name: str         # e.g., "BAAI/bge-small-en-v1.5"
    dimensions: int         # e.g., 384
    embed_texts(texts, task_type) -> list[list[float]]
    embed_query(query) -> list[float]
    dimensions -> int

The _get_provider() function lazy-instantiates the configured provider and caches it as a module-level singleton. reset_provider_cache() is used by embedding_migrate to switch providers without restart.

Search Fallback Strategy

All search tools use a two-phase strategy:

  1. Primary: Semantic search via HNSW vector index with cosine similarity. This is the main signal.
  2. Fallback: ILIKE text matching only when embeddings yield no above-threshold results. Used as a complement, never a replacement — ILIKE fills the gaps where the embedding model might miss.

This ensures reliable recall even when the embedding model's semantic understanding doesn't align perfectly with the query.
